# `.ai/orchestration.yml` — schema reference

Project-specific configuration for the orchestration fleet. Lives at the repo root. Required: every adopting OM project ships its own. `om-orchestrate init` writes a stub from the template below.

## Schema (v1)

```yaml
schema_version: 1

# E2E test runner — what command, what env, what timeout
e2e:
  command: "yarn test:integration:ephemeral"     # whatever this project calls it
  required_env:
    - OM_PRM_WIC_IMPORT_SECRET                   # project-specific; can be empty list
  timeout_minutes: 15
  artifact_paths:                                 # what to attach if tests fail
    - "test-results/"
  allow_failure_output: false                     # if true, e2e agent may include first 50 lines of failure output in PR comments. Default false (per Communication style § Secrets).

# Where things live in the repo
paths:
  run_plans: ".ai/runs"
  specs: ".ai/specs"
  app_spec: "app-spec/app-spec.md"

# Label vocabulary — prefix is configurable for projects that want a different convention
labels:
  prefix: "status:"
  human_veto: "human-review"

# How merges happen on auto-merge
merge:
  strategy: "squash"                              # squash | merge | rebase
  delete_branch: true
  base_branch: "develop"                          # or "main" — whatever the project's main integration branch is
  required_checks: []                             # gh check names that MUST be green before auto-merge

# Orchestration tunables
orchestration:
  parallel_n: 1                                    # Phase 1: 1. Phase 2 raises to 5.
  poll_cadence_seconds: 60                         # coding agent /loop tick
  e2e_poll_cadence_seconds: 30
  idle_exit_ticks: 5                               # exit after N consecutive empty polls

# Recovery thresholds (used by Phase 3; harmless in earlier phases)
recovery:
  stale_claim_minutes: 30                          # claim:* label older than this is treated as expired
  stale_e2e_minutes: 20                            # status:e2e-running older than (e2e.timeout_minutes + 5) is reset

# Telemetry — instrumented from v1.12.0
telemetry:
  enabled: true
  log_dir: "/tmp/om-telemetry"                     # per-tick cost + duration written here as jsonl
```

## Template (init writes this)

The stub `init` produces, populated from project detection. Comments preserved so the user knows what's there for them to tune.

```yaml
# .ai/orchestration.yml
# Generated by /om-orchestrate init on <DATE>.
# Edit freely; running init again will detect drift and ask before overwriting.

schema_version: 1

e2e:
  # The command to run integration / e2e tests.
  # Detected from package.json: "<DETECTED_COMMAND_OR_NONE>"
  command: "<DETECTED_OR_PLACEHOLDER>"
  required_env: []          # add env var names this project's e2e command needs
  timeout_minutes: 15
  artifact_paths: []
  allow_failure_output: false

paths:
  run_plans: ".ai/runs"
  specs: ".ai/specs"
  app_spec: "app-spec/app-spec.md"

labels:
  prefix: "status:"
  human_veto: "human-review"

merge:
  strategy: "squash"
  delete_branch: true
  base_branch: "<DETECTED_BASE_BRANCH>"
  required_checks: []

orchestration:
  parallel_n: 1             # Phase 1 starts at 1; raise after Phase 2 ships and you've measured cost
  poll_cadence_seconds: 60
  e2e_poll_cadence_seconds: 30
  idle_exit_ticks: 5

recovery:
  stale_claim_minutes: 30
  stale_e2e_minutes: 20

telemetry:
  enabled: true
  log_dir: "/tmp/om-telemetry"
```

## Validation rules

The dispatcher validates the config on every run. Fail-fast on:

- `schema_version` not 1 → abort.
- `e2e.command` empty → abort: *"Set e2e.command in .ai/orchestration.yml. Detected scripts: …"*
- `merge.base_branch` doesn't exist on origin → abort: *"merge.base_branch '<X>' not found on origin."*
- `merge.strategy` not in {squash, merge, rebase} → abort.
- `orchestration.parallel_n` < 1 or > 20 → warn; clamp.

Soft warnings (don't block):

- `e2e.required_env` items not in environment → warn at run time, not init time (init checks once; user may set after).
- `recovery.stale_claim_minutes` < 5 → warn (likely too aggressive).

## Reading the config from agent prompts

Agent prompts (coding-agent, e2e-agent) don't read YAML directly — they read env vars exported by the dispatcher. The dispatcher converts YAML to env vars on startup:

```bash
# scripts/dispatcher.sh excerpt
yq -r '.e2e.command' .ai/orchestration.yml > /tmp/.om-export-e2e-command
export OM_E2E_COMMAND=$(cat /tmp/.om-export-e2e-command)
# ...
```

Future hardening: a generated TS/JS module the prompts can `require`.

## Future fields (not in v1)

These are placeholders documented now so projects don't end up with conflicting layouts later:

```yaml
projects_v2:                # Phase 3 - GitHub Projects v2 integration
  enabled: false
  project_id: "PVT_..."
  status_field_id: "PVTSSF_..."

cost_budget:                # Phase 3 / future
  per_run_usd_max: 1000
  per_tick_alert_usd: 5

webhooks:                   # Future - if/when polling proves insufficient at high N
  enabled: false
  endpoint: ""
```

These keys are reserved. Adding them in a future schema_version will not require renaming.
